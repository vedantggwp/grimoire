## Overview

Building an MCP server involves three core steps: creating a McpServer instance, registering capabilities (tools, resources, prompts), and connecting a transport layer.

## Creating a Server

```typescript
const server = new McpServer({ 
  name: 'my-server', 
  version: '1.0.0' 
});
```

Optional server configuration includes instructions that describe cross-tool relationships and workflow patterns:

```typescript
const server = new McpServer(
  { name: 'db-server', version: '1.0.0' },
  {
    instructions: 'Always call list_tables before running queries...'
  }
);
```

## Transport Configuration

### Stdio Transport (Local Integration)

```typescript
const transport = new StdioServerTransport();
await server.connect(transport);
```

### Streamable HTTP Transport (Remote)

```typescript
const transport = new NodeStreamableHTTPServerTransport({
  sessionIdGenerator: () => randomUUID()
});
await server.connect(transport);
```

Options include enableJsonResponse for plain JSON instead of SSE and sessionIdGenerator for stateful sessions.

## Tools

Tools enable client invocation of server actions. Register with input schemas using Zod:

```typescript
server.registerTool(
  'calculate-bmi',
  {
    title: 'BMI Calculator',
    description: 'Calculate Body Mass Index',
    inputSchema: z.object({
      weightKg: z.number(),
      heightM: z.number()
    }),
    outputSchema: z.object({ bmi: z.number() })
  },
  async ({ weightKg, heightM }) => {
    const output = { bmi: weightKg / (heightM * heightM) };
    return {
      content: [{ type: 'text', text: JSON.stringify(output) }],
      structuredContent: output
    };
  }
);
```

### Tool Annotations

Signal tool behavior to clients (read-only, destructive, idempotent):

```typescript
annotations: {
  title: 'Delete File',
  destructiveHint: true,
  idempotentHint: true
}
```

### ResourceLink Outputs

Tools can return resource references without embedding content:

```typescript
const links: ResourceLink[] = [
  {
    type: 'resource_link',
    uri: 'file:///projects/readme.md',
    name: 'README',
    mimeType: 'text/markdown'
  }
];
return { content: links };
```

### Error Handling

Return isError: true for tool-level errors visible to the LLM, enabling self-correction:

```typescript
if (!res.ok) {
  return {
    content: [{ type: 'text', text: `HTTP ${res.status}` }],
    isError: true
  };
}
```

Unhandled exceptions are automatically caught and converted to error responses.

## Resources

Resources expose read-only data controlled by the host application.

Static resource:

```typescript
server.registerResource(
  'config',
  'config://app',
  { title: 'Application Config', description: 'Application configuration data', mimeType: 'text/plain' },
  async uri => ({
    contents: [{ uri: uri.href, text: 'App configuration here' }]
  })
);
```

Dynamic resources use URI patterns with discovery via ResourceTemplate.

## Prompts

Prompts provide reusable interaction templates explicitly invoked by users:

```typescript
server.registerPrompt(
  'review-code',
  {
    title: 'Code Review',
    description: 'Review code for best practices and potential issues',
    argsSchema: z.object({ code: z.string() })
  },
  ({ code }) => ({
    messages: [{
      role: 'user' as const,
      content: { type: 'text' as const, text: `Please review this code:\n\n${code}` }
    }]
  })
);
```

## Completions

Wrap schema fields with completable() to provide argument suggestions:

```typescript
language: completable(
  z.string().describe('Programming language'),
  value => ['typescript', 'javascript', 'python'].filter(lang => lang.startsWith(value))
)
```
